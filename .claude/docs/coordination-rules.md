# Правила Координации Гемблинг Студии

Успех гемблинг игры (особенно в жанре слотов) зависит от идеальной синхонизации Математики (RTP), Проектирования кода, и "Сочности" отдачи (Juiciness).

## Принципы Коллаборации

1. **Русский Язык**
   Все взаимодействия между пользователем и агентами (ответы, вопросы, логи) — на **русском языке**. На английском: только код, классы и файлы.
   
2. **Консультативный Стиль**
   Агенты не принимают самостоятельных решений без пользователя (кроме навыков `/auto-*`).
   Паттерн: *Задал вопрос -> Предложил 2-3 варианта -> Пользователь выбрал -> Написал черновик -> Пользователь утвердил -> Сохранил в файл*.

3. **Соблюдайте Субординацию**
   - Только `creative-director` меняет кор-геймплей и видение(Pillars).
   - Только `rtp-mathematician` утверждает новую модель весов и вероятностей после проверки `/balance-check`.
   - `slot-programmer` НЕ ИМЕЕТ ПРАВА захардкодить шанс выигрыша (например, `if (Random().nextDouble() < 0.1) win!`). Он обязан считывать его из `SlotConfig`/GDD.
   - `juice-artist` не делает анимацию длиннее 3-4 секунд, чтобы не замедлять Game Loop. `gambling-game-designer` утверждает длину.

## Механизмы Обработки Конфликтов

Ошибки неизбежны. Если одна механика противоречит другой, сделайте паузу и привлеките профи:

**Если код противоречит GDD:** `lead-programmer` и `gambling-game-designer` приходят к общему знаменателю. Если фича невозможна из-за архитектуры Flame, GDD обновляется.
**Если RTP ниже 90% или выше 98%:** Игра останавливается. Вызываем `rtp-mathematician`, он итерирует `weights`(веса символов), пока не попадет в окно 95-97%. Только потом `slot-programmer` обновляет код.

## Передача Работы
При передаче задачи от Математика -> Дизайнеру -> Программисту -> Спец-эффектам, используйте навык `/team-gambling`. Агент обязан передать точную ссылку на рабочие документы (например GDD `design/gdd/[file].md`) следующему агенту в цепи.
